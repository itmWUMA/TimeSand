# TimeSand 产品路线图

> 版本: v1.1 | 更新日期: 2026-04-18 | 基准版本: v0.0.1 (MVP)

## 1. 产品愿景

TimeSand 是一个自托管的智能照片墙与音乐盒。核心价值不是"管理照片"，而是**"重新发现回忆"** — 通过卡牌抽取的游戏化体验和背景音乐的氛围营造，让用户在日常中自然地回顾过往。

**北极星指标**: 用户每周主动打开 TimeSand 进行抽卡/幻灯片体验的频率。

**产品关键词**: 灵动、沉浸、仪式感、温暖

---

## 2. 现状评估 (v0.0.1)

### 2.1 已完成

- 照片上传（批量、EXIF提取、缩略图生成）
- 相册 & 标签管理（CRUD、多对多关联）
- 卡牌抽取（时间加权随机算法）
- 幻灯片播放（全屏、自动切换）
- 音乐上传 & 播放列表管理
- 音乐播放器（播放控制、循环模式）
- Docker 单容器部署
- 暗色沉浸主题（深色背景 + 琥珀/金色强调色）

### 2.2 核心问题

| 类别 | 问题 | 影响 |
|------|------|------|
| **UI/设计** | 界面生硬、不够现代化，缺乏产品定位中"灵动"的特质 | 与产品愿景脱节 |
| **UI/设计** | 无统一设计语言，TailwindCSS 默认样式堆砌 | 页面风格不一致 |
| **交互** | 微动画几乎缺失，GSAP 价值未发挥 | 交互生硬无反馈 |
| **体验** | 抽卡动画简陋，距设计稿描述的仪式感差距大 | 核心体验无吸引力 |
| **体验** | 幻灯片缺少高级转场（Ken Burns 等） | 沉浸感不足 |
| **体验** | 无 UI 音效，"音乐盒"定位名不副实 | 氛围感缺失 |
| **产品感** | 缺少亮点功能，没有让人眼前一亮的设计 | 难以留住用户 |
| **本地化** | 无 i18n，界面中英混杂 | 不专业，理解困难 |
| **内容** | 不支持 HEIC 格式（iPhone 默认格式） | 大量用户无法正常使用 |
| **引导** | 无首次体验引导，新用户不知从何开始 | 上手门槛高 |
| **UX 缺陷** | Card Draw 页显示调试文本，音乐栏显示占位文字 | 不专业 |
| **功能缺失** | Settings 页完全为空 | 配置无法调节 |
| **移动端** | 导航未折叠，无触摸手势支持 | 手机/平板不可用 |
| **数据安全** | 无备份恢复机制 | 自托管用户核心焦虑 |
| **基础设施** | 无 CI/CD，无数据库迁移 | 迭代效率低，升级有风险 |

## 3. 迭代策略

### 3.1 总体原则

```
先打磨 → 后扩展 → 再进阶
```

- **Phase 1 — 打磨期**: 设计系统驱动的整体升级，让产品达到"可以自信展示"的水平
- **Phase 2 — 增强期**: 扩展内容形态，强化核心差异化体验
- **Phase 3 — 扩展期**: 用户系统、外部集成等生态能力

### 3.2 版本策略

采用 **语义化版本** + **按需发布**:

```
v0.Y.Z
  │ │ └── patch: 修复、微调（不改功能行为）
  │ └──── minor: 功能增量或明显体验改善
  └────── major: 0.x = 早期迭代，1.0 = 首个稳定正式版
```

- 每完成一个有意义的功能集合即可发版
- 不绑定固定时间周期
- 每个版本需通过完整测试 + Docker 构建验证

### 3.3 发布标准

每个版本发布前需满足:
- [ ] 前端 lint + type-check 通过
- [ ] 后端 pytest 全部通过
- [ ] Docker 构建并启动成功
- [ ] 核心流程人工验证（上传照片 → 创建相册 → 抽卡 → 幻灯片）
- [ ] 无未替换的占位文本或调试信息

## 4. Phase 1 — 打磨期

> 目标: 以设计系统为驱动，对产品做整体视觉与交互升级，让核心体验达到"令人印象深刻"的水平。
> 预期产出: v0.1.0
> 关键理念: **不是修修补补，而是设计系统驱动的整体重塑**

### 4.1 概览

| 迭代 | 主题 | 核心产出 |
|------|------|----------|
| 1.1 | 设计系统 + i18n + HEIC | 建立视觉基础、国际化框架、iPhone 照片支持 |
| 1.2 | 核心体验重塑 | 抽卡仪式、幻灯片沉浸感、音效设计、首次体验 |
| 1.3 | 全页面升级 | 所有页面应用设计系统，达到产品级品质 |
| 1.4 | 响应式 & 移动端 | 汉堡菜单、触摸手势、移动端布局适配 |
| 1.5 | 基础设施 | CI/CD、数据库迁移、备份恢复、性能基线 |

### 4.2 迭代 1.1 — 设计系统 + i18n + HEIC

**目标**: 建立后续所有 UI 工作的基础设施——统一设计语言、国际化框架、核心格式支持。

| 任务 | 描述 | 优先级 |
|------|------|--------|
| Design Token 体系 | 定义颜色、圆角、阴影、间距、字体的 CSS 变量 / Tailwind 扩展 | P0 |
| 动效规范 | GSAP 动画预设库：缓动曲线、时长标准、入场/离场/悬浮动效模板 | P0 |
| vue-i18n 集成 | 搭建 i18n 框架，提取现有所有硬编码文本为 zh/en 语言包 | P0 |
| 语言切换 | Settings 或导航栏提供中英文切换，默认跟随浏览器语言 | P0 |
| HEIC/HEIF 支持 | 后端上传时自动转换 HEIC → JPEG，前端无感知 | P0 |
| 组件库基础 | 基于设计 Token 封装 Button、Card、Modal、Toast 等基础组件 | P1 |

### 4.3 迭代 1.2 — 核心体验重塑

**目标**: 让抽卡和幻灯片这两个核心场景达到"令人印象深刻"的仪式感，成为产品亮点。

| 任务 | 描述 | 优先级 |
|------|------|--------|
| 抽卡仪式升级 | 完整视觉旅程：背景氛围变化 + 3D 卡牌翻转/飞出/散落 (GSAP) | P0 |
| 音效设计 | 抽卡翻牌声、揭示音、幻灯片切换音、UI 微反馈音 (howler.js) | P0 |
| 回忆文案 | 抽到 "N 年前的今天" 照片时，显示温暖的时间文案 | P0 |
| 幻灯片转场 | Ken Burns（缓慢缩放平移）、淡入淡出、柔和切换 | P0 |
| 首次体验引导 | 3-4 步引导动画介绍 TimeSand 的理念和操作 | P0 |
| Demo 数据 | 内置一组示例照片和音乐，零上传即可体验抽卡和幻灯片 | P1 |
| 空状态引导 | 无照片/音乐时显示引导提示，引导用户上传 | P1 |

### 4.4 迭代 1.3 — 全页面升级

**目标**: 将设计系统应用到每个页面，清除所有 MVP 遗留的粗糙痕迹，达到统一的产品级品质。

| 任务 | 描述 | 优先级 |
|------|------|--------|
| 清除占位/调试文本 | 移除 Card Draw 调试文本、Music Player Slot 占位文字 | P0 |
| 音乐播放器集成 | 底部栏实现真实迷你播放器（进度条、封面、控制按钮） | P0 |
| Settings 页实现 | 幻灯片间隔、抽卡设置、语言切换、存储统计 | P0 |
| 照片详情查看 | 点击照片查看大图 + EXIF 信息面板 | P0 |
| 导航高亮修复 | 确保当前页面在侧边栏正确高亮 | P1 |
| 相册卡片优化 | 展示照片数量、最近更新时间等丰富信息 | P1 |
| 照片加载优化 | 缩略图懒加载、骨架屏、渐进显示 | P1 |
| 页面切换动效 | 路由切换时的柔和过渡动画 | P2 |

### 4.5 迭代 1.4 — 响应式 & 移动端

**目标**: 手机和平板上也能获得良好体验（桌面端打磨完成后进行）。

| 任务 | 描述 | 优先级 |
|------|------|--------|
| 汉堡菜单 | 移动端侧边导航折叠为汉堡按钮 | P0 |
| 抽卡手势 | 左滑抽卡、右滑撤销 | P0 |
| 照片网格响应式 | 不同屏幕宽度自适应列数 | P0 |
| 上传页移动适配 | 移动端用相册/文件选择替代拖拽 | P1 |
| 触摸优化 | 按钮尺寸、间距、点击区域适配触摸操作 | P1 |

### 4.6 迭代 1.5 — 基础设施

**目标**: 建立可持续迭代的工程基础，保障数据安全。

| 任务 | 描述 | 优先级 |
|------|------|--------|
| GitHub Actions CI | PR 自动运行 lint + type-check + pytest + Docker build | P0 |
| 数据库迁移 | 引入 Alembic，支持版本间的 schema 升级 | P0 |
| 备份与恢复 | 一键导出（数据库 + 照片 + 音乐）、一键恢复 | P0 |
| Release 自动化 | tag → build Docker image → create GitHub Release | P1 |
| 错误处理 | 前端全局 Toast 提示、后端统一错误响应格式 | P1 |
| 性能基线验证 | 虚拟滚动（照片网格）、缩略图缓存策略、数据库索引优化 | P1 |
| 日志规范 | 后端结构化日志，便于排查问题 | P2 |

## 5. Phase 2 — 增强期

> 目标: 扩展内容形态，强化"重新发现回忆"的核心差异化体验。
> 预期产出: v0.2.x ~ v0.4.x

### 5.1 回忆发现增强

| 功能 | 描述 |
|------|------|
| "今日回忆"模块 | 首页展示"N年前的今天"照片合集，主动推送回忆 |
| 抽卡历史 | 记录抽卡历史，可回顾过往抽到的照片 |
| 收藏机制 | 标记喜欢的照片，收藏夹功能 |
| 时间线视图 | 按时间轴浏览照片，感受时间流逝 |
| 地图视图 | 基于 EXIF GPS 数据在地图上展示照片分布 |
| 回忆卡片分享 | 抽卡结果生成精美图片（照片 + 日期 + 文案 + 水印），可保存/分享 |

### 5.2 内容形态扩展

| 功能 | 描述 |
|------|------|
| 短视频支持 | 上传和管理短视频，抽卡可抽到视频（自动播放），幻灯片穿插视频 |
| 照片去重 | 基于文件哈希检测重复照片，上传时提示 / 批量清理 |

### 5.3 照片管理增强

| 功能 | 描述 |
|------|------|
| 批量操作 | 多选照片 → 批量加入相册 / 打标签 / 删除 |
| 搜索功能 | 按文件名、标签、日期范围搜索照片 |
| 照片编辑基础 | 旋转、裁剪等基础操作 |
| 智能分组 | 按拍摄地点、日期自动分组建议 |

### 5.4 音乐体验增强

| 功能 | 描述 |
|------|------|
| 播放器 UI 升级 | 更精致的播放器界面，封面展示 |
| 场景联动 | 抽卡/幻灯片场景自动切换对应播放列表 |
| 音量渐变 | 场景切换时音乐平滑过渡 |

### 5.5 展示与主题

| 功能 | 描述 |
|------|------|
| 亮色主题 | 柔和的浅色模式，适合白天使用场景 |
| 瀑布流视图 | 类 Pinterest 的照片墙展示模式 |
| 数码相框模式 | 专为壁挂屏/平板设计的全天候展示模式 |
| 屏保模式 | 无操作时自动进入幻灯片 |

> 预期产出: v0.5.x+，远期目标 v1.0

| 功能 | 描述 |
|------|------|

| 多用户支持 | 家庭成员各自的相册和配置 |
| 访客模式 | 无需登录即可浏览共享相册 |

### 6.4 平台化

| 功能 | 描述 |
|------|------|
| PWA 支持 | 可安装为桌面/移动端应用 |

## 7. 迭代开发流程

### 7.1 流程总览

在 MVP 阶段验证的流程基础上，增加**迭代规划**和**回顾**环节:

```
┌─────────────────────────────────────────────────┐
│                  迭代周期                         │
│                                                   │
│  迭代规划 → [需求对齐 → 任务分解 → 并行开发       │
│              → 集成测试 → 发布] → 迭代回顾         │
│                                                   │
│  ← ─ ─ ─ ─ ─ 按需发布，不绑定固定周期 ─ ─ ─ ─ → │
└─────────────────────────────────────────────────┘
```

### 7.2 迭代规划

每个迭代开始前:

1. **确定迭代范围** — 从路线图中挑选当期要完成的功能集合
2. **梳理依赖关系** — 确认任务间的前后依赖
3. **定义验收标准** — 每个功能的完成标志是什么
4. **评估风险** — 识别技术难点或不确定项

产出: 迭代计划文档 (`docs/tasks/<iteration-name>.md`)

### 7.3 执行流程 (沿用 MVP)

```
需求对齐 (Claude Code)
    ↓ 输出: docs/superpowers/specs/<spec>.md
任务分解 (Claude Code)
    ↓ 输出: docs/tasks/<plan>.md + docs/tasks/<plan>-NN-<task>.md
并行开发 (Codex / Claude Code)
    ↓ 各 feat/<task-slug> 分支独立开发
集成 (Claude Code)
    ↓ feat → dev (PR) → 冲突解决 → 验证
发布
    ↓ dev → main (PR) → tag → release
```

### 7.4 迭代回顾

每个版本发布后:

- 记录本次迭代中遇到的问题和经验
- 更新路线图（如有调整）
- 更新进度追踪文档

### 7.5 分支策略 (沿用)

```
main          ← 只接受 dev 的合入（生产就绪）
  └── dev     ← 集成分支，接受 feat/* 的 PR
       └── feat/<task-slug>  ← 功能分支
```

### 7.6 进度追踪

- **路线图级别**: 本文档 (`docs/product-roadmap.md`) — Phase/迭代完成状态
- **迭代级别**: `docs/tasks/<iteration-name>-progress.md` — 各任务完成状态
- **每次发版后更新路线图中的完成标记**
